Professional Summary
O2O 여행 플랫폼 창업멤버를 시작으로, 블록체인 거래소·모빌리티 그리고 커머스(원앱)까지 — 새로운 환경에서도 즉시 팀을 구성해 퍼포먼스를 만들어온 그룹사의 핵심 0 to 1 프로젝트마다 플랫폼을 직접 설계·성장시켜온 실행형 PO입니다.
제게는 일관되게 유지해온 설계 원칙이 하나 있습니다. 유저의 흐름과 관성을 활용하는 설계. 유저를 억지로 움직이려 할 때 서비스는 실패하고, 유저가 이미 가려는 방향에 다음 행동을 얹을 때 지표가 움직인다는 것을 반복된 실험과 실패를 통해 확인해왔습니다. 마지막 커머스 플랫폼에서 광고 CTR 0.36%→10%+ 월매출 66배, 메인 개편 후 인당 챌린지 참여율 +141%는 모두 이 원칙을 데이터로 검증한 결과입니다.
복잡한 금융 개념 앞에서 유저가 멈추는 지점이 어디인지, 가고 싶은 곳은 어디인지 — 그 흐름을 읽고 설계하는 것이 증권 도메인에서 제가 기여할 수 있는 지점입니다. 도메인 지식은 빠르게 쌓을 수 있고, 유저의 흐름을 읽는 감각은 9년의 다양한 도메인에서의 경험과 실패가 만들어준 것입니다.

앱테크 유저를 커머스로 전이시키는 것이 플랫폼의 핵심 BM 과제였습니다. "미션이라는 친숙한 포맷이 커머스 진입 거부감을 낮출 것"이라는 가설로 웰컴미션을 설계했고, 여러 기획전을 구경할 수 있도록 설계된 미션을 통해 커머스 방문율은 10배 올랐습니다. 하지만 구매 전환은 일어나지 않았습니다.

데이터를 들여다보니 원인은 명확했습니다. 가입 직후 쿠폰 하나로 구매까지 이어지는 것은, 유저 입장에서 초대형 허들이었습니다. 유저의 흐름을 억지로 꺾으려 했던 것이 문제였습니다.

가설이 틀렸음을 인정하고 팀과 논의 후 구조를 바꿨습니다. 쿠폰 대신 포인트를 모아가는 여정으로 메인 흐름을 재설계했습니다. 유저가 이미 가려는 방향 — 포인트를 쌓는 행동 — 위에 구매 시나리오를 얹었습니다. 유저의 흐름을 거스르지 않자, 구매 사이클이 3개월에서 1.8개월로 단축됐습니다. 서비스가 행동 패턴을 바꾼 것이 아니라, 행동 패턴 위에 서비스를 맞춘 것입니다.

TF 구성 배경
기존 스쿼드 구조로는 메인 개편의 속도와 집중도를 확보하기 어렵다고 판단. 기획·디자인·데이터 직군을 묶은 전담 TF를 직접 제안·구성하고 TF장으로서 목표 설정과 운영 전반을 리딩. 핵심 지표 달성 후 TF 종료. 작업 우선순위는 트래픽 임팩트 → 유저 블로킹 여부 → BM 영향력 → 개발 난이도 순으로 판단했고, 전략이 다양하면 A/B(필요시 A/B/C)로 가설을 검증해 결론을 냈습니다.
가설
메인화면에서 챌린지로의 진입 마찰을 줄이고 서비스 정체성을 명확히 하면, 체류시간과 챌린지 참여도가 동시에 오른다.
성공 판단 기준
A/B 테스트에서 챌린지 참여율과 체류시간 두 지표가 동시에 유의미하게 상승하는 경우를 성공으로 정의. 둘 중 하나만 오르는 경우는 구조 개선이 아닌 일시적 효과로 판단. Airbridge 이벤트 데이터와 Snowflake 대시보드로 직접 모니터링.
Learning

가설 (1차)
미션이라는 친숙한 포맷이 커머스 진입 거부감을 낮출 것이다.
실패 원인 및 피벗
쿠폰 전환율 저조. 가입 직후 구매는 유저에게 여전히 초대형 허들임을 데이터로 확인. 단발 보상(쿠폰) → 포인트 적립 여정으로 구조 재설계.
다음 사이클 (2차 가설 및 결과)
가설: 포인트 적립 시간이 단축되면 구매 전환이 더 빨라진다 → 할인형 챌린지 기획.
결과: 기존 앱테크 유저는 충분한 포인트 없이는 지출 자체에 저항이 있음을 재확인. 단, 신규 유입 고객에게는 할인 보상이 유효했고, 결과적으로 챌린지는 기존 유저용·신규 유저용 듀얼 구조로 운영됐습니다.
부작용: 듀얼 운영으로 메인화면의 메시지가 다시 불명확해지기 시작 — 2차 메인 개편을 준비하던 시점에 팀을 이동했습니다.
Learning
유저 관성을 꺾으려 한 것이 실패 원인. 관성을 이용해 단계별 성공 경험을 쌓게 하는 구조가 서비스 피벗의 핵심. 단, 유저 기반의 지출 심리가 BM 설계의 전제조건이 돼야 한다.

가설
지면 수 확장보다 저효율 지면 정리를 통한 유저 노출 최적화가 CTR과 총 수익을 동시에 끌어올린다.
성공 판단 기준
지면 축소 후 CTR과 월매출이 동시에 상승하는 경우를 성공으로 정의. 지면 감소에도 수익이 오르면 유저 노출 최적화 전략이 유효한 것으로 판단. 애드팝콘 CTR 데이터와 월매출 수치를 직접 비교해 판단.
Learning
효율 역산 없는 지면 확장은 유저 UX 저하와 지표 노이즈를 동시에 유발. 데이터 기반으로 제거하는 결정이 유저 경험과 수익을 함께 키운다.

서비스 간 Cross-Activation — 만보기→경매 유저 흐름 연계
경매 참여자 +72.9% 입찰건수 +20.8%
플랫폼에서 가장 많이 쓰이는 기능인 만보기가 보상 획득 이후 추가 액션 없이 끊기는 것을 트래픽 연계의 핵심 병목으로 직접 정의했습니다. 가장 트래픽이 많은 진입점에 연계 경로를 붙이면 Cross-Activation이 일어난다는 가설로 만보기 → 경매 경로를 설계했습니다. 초기엔 챌린지 간 강제 연결도 시도했지만, 유저는 본인이 원하는 것을 하려는 의지가 강해 아무거나 제안하면 무시된다는 것을 데이터로 확인했습니다. 유저가 자연스럽게 가려는 방향에만 연계점을 놓는 방식으로 재설계했습니다 — 흐름을 거스르지 않자 경로가 작동했습니다.
가설
트래픽이 가장 많은 기능(만보기)에 연계 경로를 붙이면 별도 마케팅 없이 Cross-Activation이 일어난다.
실패 및 피벗
챌린지 간 순차 연결 시도 → 유저는 제안된 다음 챌린지가 아닌 본인이 원하는 것으로 이동. "연계 경로는 유저의 자연스러운 다음 행동에 맞닿아야 한다"는 원칙으로 재설계.
성공 판단 기준
적용 전후 7일 Airbridge 데이터 직접 비교. 만보기 유입 유저의 경매 참여·입찰 행동을 직접 추적. 학습 효과로 초기 대비 소폭 하락했으나, 트래픽 연계 관점에서 구조적 경로가 작동했음을 확인해 성공으로 판단.
Learning
Cross-Activation은 경로의 수가 아니라 유저 행동 흐름과의 정합성이 결정함. 플랫폼이 원하는 경로가 아닌 유저가 자연스럽게 가려는 방향에 연계점을 놓아야 한다.

온보딩 자동화
차량 등록 완료율 50% 정체를 성장 병목으로 직접 정의하고 원인을 분석했습니다. 가입과 차량등록이 분리된 구조가 심리적 허들임을 데이터로 확인했습니다. "차량번호 하나만 입력하면 나머지는 자동으로" — 단 하나의 원칙으로 플로우를 재설계했습니다. 잔여 30% 이탈은 타명의 차량 등록 수요로 파악했으나, 현행 정책 구조상 수용 불가로 판단해 차기 과제로 정의했습니다.
완료율 50% → 70% (+20%p) 가입 소요시간 28% 단축 수기 심사 50% 감소
가설
사용자들이 앱을 설치한 목적(차량 등록 후 단속알림 수신)을 가장 먼저 노출하면 온보딩 완료율이 유의미하게 오를 것이다.
한계 및 다음 과제
잔여 30% 이탈 원인 = 타명의 차량 등록 수요. 현행 정책·시스템 구조상 수용 불가로 판단, 차기 정책 설계 과제로 정의.

설계 과정 상세 — 문제 분석 · 설계 전략 · 구간별 설계 의도
문제 분석
CS 히스토리·탈퇴 사유 등 내부 DB를 분석해 이슈를 도출. 핵심 병목 두 가지:
① 외부 본인인증창(표준인증창)에서 이탈률 10% 발생
② 가입 완료 후 차량등록까지 유저가 직접 다음 행동을 탐색해야 하는 단절 구조.
설계 전략
단순히 이탈률을 줄이는 것이 아니라, 온보딩 범위 자체를 확장하는 방향으로 재정의. 기존 온보딩 구간(회원가입)에 차량등록·지역설정까지 포함시켜, 유저가 서비스 핵심 가치에 도달하는 시점까지를 하나의 플로우로 설계.
구간별 설계 의도
가입 구간 — 본인인증을 소켓 방식으로 내재화(이탈 10%→4%). 로그인 정보 없을 경우 가입으로 즉시 연결(둘러보기 가능), 신규·기존 가입자 분기 처리.
등록 구간 — 4→2 depth 축소. 소유주 확인 API 먼저 체크하여 심사 등록 사전 차단, 내부 자동차 DB 활용하여 법인차량 오탈자 자동 방지. API 장애 시 fallback 시나리오 별도 설계.
완료·설정 구간 — 완료 화면에서 지역설정으로 자연 유입되는 UI 배치. 불필요한 지역 설정 해제로 오발송률 절감

단속알림 인증제 · 데이터 기반 MAU 리텐션 설계
휘슬은 본질적으로 보험성 서비스입니다. 한 번 설치하면 재방문 이유가 없는 구조가 MAU의 핵심 병목이었습니다. 보험성 서비스의 MAU를 끌어올리기 위해선 유저가 앱을 열 이유를 만들어야 했습니다. 이벤트성 알림으로 단순 접속을 유도하는 방법도 있었지만, 월 1회 이상 정기 재방문을 유도하는 구조적 미션이 필요하다고 판단했습니다. 데이터를 분석하니 차량 소유자들이 가장 민감하게 반응하는 것은 '단속' 자체가 아니라 '내 차 정보가 어디 쓰이는가'였습니다. 이 심리를 활용해 단속알림 인증제를 '개인정보 보호' 프레임으로 재패키징했고, 민원 없이 전 회원 대상 MAU 견인 장치로 정착시켰습니다. 유저가 이미 갖고 있는 불안의 흐름 위에 행동을 얹은 설계였습니다.
MAU 인입률 15% 이상 발송 비용 절감 (문자 → 카카오 채널 전환)
가설
이벤트성 알림이 아닌 '개인정보 보호' 심리를 활용한 정기 인증 구조를 만들면, 보험성 서비스의 월 1회 이상 재방문을 구조적으로 확보할 수 있다.
성공 판단 기준
인증제 도입 후 MAU 내 정기 재방문 유저 비율 증가를 성공으로 정의.
민원 발생 없이 전 회원 대상 적용이 안착된 시점을 구조적 성공으로 판단.
발송 비용 절감은 부수 목표로 함께 달성.
Learning
단순 이벤트 알림이 아닌 유저의 심리(정보 불안)를 구조적 재방문 동기로 전환한 것이 핵심. 리텐션 설계는 앱을 열 이유를 만드는 것이 아니라, 유저가 이미 갖고 있는 감정 흐름 위에 행동을 얹는 것이다.

(현대)제휴 기능 모듈화 · 그룹사 자산화
유료 마케팅 없이 지속 가능한 신규 유저 유입 구조를 만드는 것을 목표로 설정했습니다. 현대차(마이현대) 제휴 연동 가입을 구축해 비용 없이 일평균 200~400명의 신규 가입을 확보했습니다. 여기서 멈추지 않고 단속알림 서비스 자체를 독립 모듈로 설계했습니다. 이 모듈은 이후 그룹사 여행 관련 서비스 '여가생활안심보호'에 그대로 이식되어 재적용됐습니다. 하나의 기능을 플랫폼 내 자산이 아닌 그룹사 전체가 재사용 가능한 인프라로 확장한 설계였습니다.
일평균 신규 가입 200~400명 무비용 모듈 그룹사 타 서비스 재적용 (여가생활안심보호)
접근 방식
그룹사 네임밸류 활용, 마이현대 제휴 채널을 통한 제로 비용 유입 경로 설계. 단속알림 기능을 서비스 종속 구조가 아닌 독립 모듈로 설계하여 그룹사 내 재사용 가능한 자산으로 구조화.
성공 판단 기준
제휴 채널을 통한 신규 가입이 유료 마케팅 없이 지속적으로 발생하는 것을 성공으로 정의. 추가로 모듈이 타 서비스에 실제 적용됐을 때 재사용 가능한 자산으로서의 가치를 확인.
Learning
제휴 설계의 핵심은 유입 경로 확보에 그치지 않는 것. 기능을 처음부터 독립 모듈로 설계하면 단일 서비스의 성과가 그룹사 전체 자산이 된다 — 마케팅 비용 없이 만든 유입 구조가 타 서비스에 그대로 이식된 것이 그 증거.

커뮤니티 플랫폼 성장 전략
초기 커뮤니티의 seeding 작업은 서포터즈 운영을 활용했습니다. 하지만 서포터즈 운영만으로는 커뮤니티 트래픽 성장에 한계가 있었습니다. 씨드 콘텐츠 전략으로 전환했지만, 처음 집중했던 차량 정보성 콘텐츠는 반응이 없었습니다. 데이터를 들여다보니 전체 이용자의 30%가 부산 등록 차량이었습니다. 이 데이터를 기반으로 '부산 교통문화'라는 주제의 글을 금요일 저녁에 의도적으로 배포했고, 주말 내내 사용자 간 인터랙션이 폭발했습니다. 유저가 원하는 것은 정보가 아니라 자신의 이야기였습니다. 이후 커뮤니티 플랫폼을 단순 게시판이 아닌 감정 기반 트래픽 엔진으로 재정의하고 운영했습니다.
방문자 3.8배 (25만 → 95만) 광고 매출 3배 성장
가설
정보성 콘텐츠보다 유저 감정을 건드리는 콘텐츠가 커뮤니티 참여를 폭발시킨다.
실패 및 피벗
초기 차량 정보성 콘텐츠: 볼거리 제공으로 트래픽을 끌 것이라 가정했으나 반응 없음. 유저 데이터(부산 등록 30%) 기반으로 지역 감정을 자극하는 콘텐츠로 피벗 → 즉각적인 인터랙션 폭발 확인.
성공 판단 기준
트래픽 수치(방문자 3.8배) 달성은 성공으로 판단. 단, 유저가 자발적으로 콘텐츠를 생산하는 단계까지는 유도하지 못했음 — 씨드 콘텐츠 의존 구조에서 벗어나지 못한 것을 미달성 과제로 정의.
Learning
커뮤니티의 핵심 변수는 콘텐츠 퀄리티가 아니라 유저의 감정 개입 여부. 트래픽을 끌어올리는 것과 자발적 생산자를 만드는 것은 별개의 과제이며, 후자를 위한 별도 설계가 필요했음.
커뮤니티 설계 구조 — 트래픽 → 데이터 → 수익의 선순환
커뮤니티를 트래픽 엔진 + 데이터 자산으로 설계 — 행태 DB 기반 맞춤 콘텐츠·타겟 광고 선순환 구조

운영비 절감 · API 자동화
서비스가 성장할수록 운영 고정비가 함께 늘어나는 구조를 인식하고, 운영비 절감이 곧 성장 여력이라는 관점에서 직접 비용 구조를 분석했습니다. 비용 병목이 되는 항목을 식별해 자동화·직연동·구조 개선으로 순차 교체했고, 총 월 약 2,600만 원을 절감했습니다.
원부 API 월 1,500만 원 절감 하이패스 직연동 월 900만 원 CDN 구조 변경 월 200만 원 총 월 약 2,600만 원 절감
① 등록원부 API 자동화 — 월 1,500만 원 절감
차량 등록 시 소유주명을 정확히 입력해야 정상 등록되는 구조로, 법인차량은 철자 오탈자로 인한 오등록이 빈번했습니다. 내부 DB에 기존 등록된(사실확인 완료) 소유주명을 재활용해 자동 매칭·수정 처리하도록 로직을 변경, 수기 심사를 대폭 제거했습니다. 소유주 변동 확인 API는 기존 일괄 처리 방식에서 평균 소유주 변경 기간을 반영한 정책으로 전환 — 등록원부 등록일 기준 최소 3개월·최대 6개월 사이 1회 처리로 변경해 불필요한 호출을 줄였습니다.
② 하이패스 직연동 — 월 900만 원 절감
하이패스 미납요금 조회 API를 외부 중계사를 통해 이용하던 구조에서, 한국도로공사와 직접 계약으로 전환했습니다. 영업 부서에 직접 의견을 개진해 추진한 케이스로, 중계 수수료를 제거해 조회 비용을 절감했습니다.
③ CDN 구조 개선 — 월 200만 원 절감
커뮤니티 트래픽 증가로 CDN 비용이 함께 늘어나는 구조였습니다. 콘텐츠 호출 시 매번 새로 불러오던 방식에서, 변경된 콘텐츠에 대해서만 호출하는 1차 캐시 처리를 추가해 불필요한 트래픽을 줄였습니다.
